Guía de introducción a GeoTools para creadores de soluciones (versión 1.0)

Requisitos:

Este documento introduce algunos elementos de programación en Java y, mientras es posible que lo aprendas sobre la marcha, quizás sería mejor buscar un buen tutorial si no tienes ninguna experiencia previa de Java (al menos, sería útil saber crear un applet).

También sería útil haber ojeado la guía para usuarios finales que sirve como una introducción rápida a los applets de GeoTools.

Lo que necesitas:

Deberías descargar la copia más reciente del paquete de demos y ejemplos de GeoTools que se encuentra en un fichero llamado demos.zip. En el momento de escribir estas líneas, es la versión 1.1 y se puede descargarlo desde la página de descarga de GeoTools.

Descarga una copia del paquete 'geotools-docs' más reciente (incluso si lleva la extensión 'dev'). Si añades http://geotools.sourceforge.net/docs a tus favoritos, siempre podrás encontrar una copia en línea de la documentación API más reciente. Sin embargo, te advertimos que a veces pueden ser documentadas ciertas características que ya no estan disponibles en el paquete de 'ejemplos y demos' que has descargado.


También tienes la opción de descargar una copia del código fuente principal de GeoTools. Incluso si no piensas modificarlo todavia, puede servir como una referencia muy útil cuando la documentación API resulte demasiado confusa para ayudarte. A veces, se pueden encontrar dos versiones del codigo fuente: una versión normal y una versión para desarrolladores (dev). De momento, es mejor descargar la que tenga la fecha más reciente.

También necesitarás un compilador de Java (versión 1.1 o superior).

Por fin, necesitarás un navegador con soporte de Java 1.1: por ejemplo, Netscape 4.06 o superior / IE5 o superior. Si decargas el navegador, ten en cuenta que deberías elegir la opción que incluye una máquina virtual Java (Java Virtual Machine), ya que en algunas de las versiones recientes no se incluye por defecto).

Como compilar un applet de GeoTools:

Descomprime el archivo demos.zip y abre el fichero readme.html. Echa un vistazo a cada ejemplo (haz clic en los enlaces). Para ver más detalles, echa un vistazo a la guía de introducción para usuarios finales.

Si abres la carpeta de demostraciones, encontrarás todos los ficheros y subdirectorios que están utilizados por todos los demos con los que acabas de jugar. Por ejemplo, el primer demo utiliza:

Example1.html
Example1.class
jars\geotools.jar
jars\collections.jar
jars\support.jar
maps\statepop.zip

Además de estos ficheros, hay un fichero llamado Example1.java. Éste es el que nos interesa ahora, ya que contiene el código fuente del applet utilizado en Ejemplo1.

El primero que debes hacer es intentar compilar el applet, sin hacerle ninguna modificación. Si todo va bien y no ves ningún error, podemos mirar como funciona el código fuente.

Todo depende de tu entorno de desarrollo Java, pero es posible que tengas que cambiar la sentencia siguiente (consulta la documentación de tu compilador para mayor información).

javac -classpath jars/geotools.jar:jars/collections.jar Example1.java


Con un poco de suerte, no aparecerá ningun mensaje de error, pero si no es así y no entiendes por qué, haz una copia de los mensajes y envía una solicitud de soporte (elige "getting started" de la lista de opciones). Cuantos más detalles nos facilite sobre los errores, más fácil nos será que te ayudemos. Si nos envias información adicional sobre el sistema operativo y el entorno de desarrollo Java que utilizas, incluso mejor.

Como prueba final, intenta ejecutar el applet a ver si todavia funciona. O bien abre Example1.html en tu navegador o utiliza 'appletviewer'.

appletviewer Example1.html

Miremos el código fuente de Ejemplo 1:

Abre Example1.java en tu editor de textos favorito y mira el código. Con un poco de suerte, los comentarios del código te darán algunas pistas sobre lo que sucede pero, de todas formas, trataré los puntos principales ahora (no incluyo los comentarios aquí por razones de concisión).

import java.applet.*;
import java.awt.*;
import java.net.*;
import java.io.*;
import uk.ac.leeds.ccg.geotools.*;
import uk.ac.leeds.ccg.widgets.ToolBar;

Las primeras cuatro líneas contienen sentencias de importación estándards de Java. Sin embargo, las últimas dos importan clases específicas a GeoTools: la principal biblioteca de GeoTools y una clase llamada 'ToolBar' (barra de herramientas) de un paquete 'widgets' de soporte.

public class Example1 extends Applet
{
Viewer view = new Viewer();


La clase Viewer es el componente en el que se visualiza el mapa. En este caso, se construye una instancia de Viewer llamada view. Como es lógico, es una de las clases más importantes de GeoTools y, si tienes el código fuente disponible, verás que también es una de las más complejas.

De momento, no es necesario conocerlo a fondo, basta con saber que facilitará el viewport (subdivisión de pantalla) en el que el mapa se visualizará.

public void init(){
setLayout(new BorderLayout());
add(view,"Center");
ToolBar tools = new ToolBar(view);
add(tools,"South");
}


El método init de un applet se llama antes de que el applet se visualice, así que el código que ves aquí coloca los componentes del applet.

Fíjate que se utiliza un BorderLayout y el objeto view se añade al centro. Ésta es una convención muy útil ya que le permite al mapa redimensionarse hasta ocupar todo el espacio disponible.

Ahora, ves como el objeto ToolBar (tools) se construye. Toma un 'Viewer' como argumento. Los 'widgets' de ToolBar facilitan unos mandos muy útiles que permiten interactuar con los viewers, así que al momento mismo de crearse, tienen que saber qué viewer controlarán. No hay nada milagroso en el 'widget' de ToolBar y resulta bastante fácil ignorarlo y crear tus propios mandos de interfaz si estás familiarizado con el AWT. Es posible que el código fuente de ToolBar te de algunas pistas.

public void start(){
try{
loadMaps();
}
catch(IOException e){
this.showStatus("Error loading map file "+e);
}
}


En cuanto se carga el applet y termina el método init(), se llama el método start(). Este método no hace nada más que llamar a otro método que carga todos los datos cartográficos.

Se utiliza el bloque try/catch para capturar cualquier tipo de excepción que pueda occurir durante el proceso de carga (puede que falte un fichero o exista un fichero corrupto).

public void loadMaps() throws IOException{
String shapefile = this.getParameter("shapefile");
URL url = new URL(getCodeBase(),shapefile);

ShapefileReader sfr = new ShapefileReader(url);

Theme t = sfr.getTheme();
view.addTheme(t);
}


Éste es el método que carga los mapas y es invocado por el método start. El primero que hace es determinar el nombre y la posición del mapa a cargar (en este caso, se obtiene la información desde una etiqueta 'param' del fichero html que acompaña al applet). Si miras el código fuente de Example1.html, encontrarás una línea que dice <param name="shapefile" value="maps/statepop">. Es el valor de maps/statepop que se utiliza aquí. Una vez recogida la información, se utiliza para construir un URL completo.

Ahora, se crea un ShapefileReader (sfr) para sacar información del shapefile denominado. GeoTools contiene un buen número de clases 'Reader' que funcionan con varios formatos de fichero. Sin embargo, en el momento de escribir estas líneas, la clase ShapefileReader es la mas estable y bien soportada.

A continuación, el ShapefileReader se utiliza para construir un objeto 'theme' (t) a partir de los datos de que dispone. En GeoTools, los 'Themes' contienen toda la información relacionada con un conjunto determinado de entidades geográficas, incluyendo los colores, los datos de atributo, las etiquetas y las entidades mismas. En realidad, los Themes no almacenan esta información de manera directa, sino que utilizan una colección de otros objetos, entre los cuales se destacan una capa (Layer) que contiene la geografía misma, un objeto GeoData que contiene la información de atributo, un Shader (sombreador) que convierte los valores GeoData en colores para efectuar el sombreado, un ShadeStyle (que controla como se pintan las entidades), y unos cuantos más.

Sin embargo, lo importante es que ShapefileReader esconde todo esto y proporciona opciones predeterminadas a todo lo arriba mencionado con la excepción del objeto Layer que es lo que construyó del shapefile mismo.

Como verás en los demos posteriores, hay más opciones 'getTheme' y éstas ofrecen más control sobre como se construye el Theme, pero aún así, puedes llamar métodos sobre el objeto theme mismo para cambiar su configuración.

Por fin, el theme (t) se añade al objeto 'view'. Hecho eso, todo lo demás está en automático. GeoTools trazará el mapa a escala, mientras el 'widget' de ToolBar maneja la selección de modos de 'zoom' y 'pan' para el viewer.

Los demás métodos del applet sólo facilitan información sobre el applet mismo y no vienen al caso.

Es importante notar aquí que el fichero que aparece en el documento html no lleva extensión alguna. Esto es porque la extensión podría ser una de varias. Generalmente, los shapefiles están compuestos de uno, dos, tres o más ficheros separados: un fichero .shp que contiene las entidades, un fichero .dbf que contiene los datos de atributo y un fichero .shx que funciona como un índice. Ficheros adicionales proporcionan más información de índice pero no se necesitan (de hecho, GeoTools puede funcionar sin el .shx y incluso sin el .dbf). GeoTools da al usuario la opción de publicar estos ficheros en formato comprimido así que, al no especificar una extensión, GeoTools buscará primero un fichero .zip y luego todos los ficheros individuales.

Miremos el código de Ejemplo 2

No trataré todo el código aquí ya que casi todo es ídentico al ejemplo anterior, así que sólo trataré las nuevas líneas del método loadMaps, que es el único que cambia.

String tooltip = this.getParameter("tooltip");
GeoData tips = sfr.readData(tooltip);
t.setTipData(tips);

Ejemplo 2 añade tooltips al mapa (tooltips son los pequeños carteles de ayuda que aparecen en la pantalla y proporcionan información sobre algún que otra característica cuando el puntero se deja sobre ella).

Para crearlos, se necesita una etiqueta 'param' adicional en el fichero html del applet. Se debería darle el nombre de una columna del fichero .dbf.

Si no sabes qué columnas existen en tu fichero .dbf, tienes algunas opciones. O bien podrías abrir el fichero .dbf dentro de una aplicación (Excel, StarOffice, o incluso ArcView si tienes una copia), o bien podrías utilizar el comando appletviewer para ejecutar Ejemplo 1 (si haces esto, te debe aparecer en la pantalla algo parecido a lo siguiente:

Found .dbf in zip file
Loaded as STATE_NAME
Loaded as STATE_FIPS
Loaded as SUB_REGION
Loaded as STATE_ABBR
Loaded as LAND_KM
Loaded as WATER_KM
Loaded as PERSONS
Loaded as FAMILIES
...


Cada una de estas líneas representa una columna del fichero .dbf.

Se utiliza el 'shapefile reader' para extraer la columna denominada del fichero .dbf y empaquetarla en un objeto GeoData, los contenidos del cual se utilizan para crear los 'tooltips'. En realidad, GeoData es una interfaz muy sencilla que define un fuente de datos capaz de enlazar valores ID con valores de datos. En la mayoria de los casos, esto quiere decir que cada entidad de una capa tiene un ID (identificación) que puede enlazarse con un valor de datos a través de llamadas a un objeto GeoData. Normalmente, el enlace es transparente ya que el objeto Theme se encarga de ello (que es lo que pasa en la última línea del nuevo código). Se llama el método setTipData(GeoData) de 'theme' para almacenar la información de los tooltips (el 'tip data') junto con las entidades geográficas.

A parte de estas tres líneas y el cambio que se hace al fichero html, no hacen falta más modificaciones.

String shadeby = this.getParameter("shadeby"); //Este valor debería corresponder al nombre de una columna del fichero .dbf

Se necesita otra etiqueta 'param': esta vez para declarar de qué columna se debería extraer los datos relacionados al sombreado.

GeoData shadeData = sfr.readData(shadeby);

Los datos se extraen del mismo modo en que se extrajeron los datos 'tooltip' del ejemplo anterior.

Shader shade = new ClassificationShader(shadeData,6,ClassificationShader.QUANTILE,Color.green,Color.orange);

Los objetos 'Shader' representan otro componente importante de GeoTools. La 'Shader interface' (interfaz de sombreo) es, en realidad, una interfaz muy sencilla que declara que las instancias deberían poder devolver un color por cualquier valor. El ClassificationShader es una implementación bastante compleja del Shader Interface que puede dividir una serie de números en 'bins' y vincular un color con cada 'bin'. En este ejemplo, crea los 'bins' a partir de unos datos que se encuentran en shadeData (se crearán seis 'bins' a partir de un algoritmo quantile). El primer bin será de color verde, el último bin se sombreará de color naranja, mientras que los demás bins serán de colores intermedios).

Theme t = sfr.getTheme();

El objeto 'theme' se construye tal como se hizo antes.

t.setShader(shade);
t.setGeoData(shadeData);

Estas dos líneas configuran el 'theme' para que utilice el shader (sombreador) construido antes y para que utilice el objeto shadeData para valores de entidades. Es importante tener en cuenta que no basta con pasar shadeData al ClassificationShader arriba mencionado. También se debería pasarlo al 'theme' como ilustrado.

Hay una serie de razones que explican lo anterior. Una de ellas es que no todos los shaders necesitan un objeto GeoData para construirse. Otra es que no es obligatorio pasar el mismo objeto GeoData al shader que al theme (aunque, en la mayoria de los casos, es precisamente lo que harás).

keys.add(shade.getKey());

Por fin, pides la llave (Key) al shader y la añades al panel que construimos antes.

¿Y ahora qué?

Bueno, ahora sabes compilar un applet y sabes como funcionan tres applets sencillos. ¿Ahora qué haces?

Pues, de momento hay dos ejemplos más en el paquete de demos (aunque Ejemplo 4 sólo ilustra como se añade un widget de 'zoom control' sencillo). De otro lado, GraphApplet añade toda una serie de características y, lo que es muy importante, introduce el concepto de HighlightManagers.

GraphApplet debería tener su propio tutorial, pero, hasta que se escriba, a ver si llegas a entenderlo lo a partir del código fuente.

Si buscas algunos ejercicios prácticos, podrías intentar modificar el programa para que se carge más de un mapa al viewer, o podrías intentar hacer funcionar otro de los shaders (sombreadores). Vale la pena echarles un vistazo a MonoShader y HSVShader.

Si te gusta aprender al estudiar el codigo fuente, echa un ojo a la lista de demos en línea ya que un buen numero de estos tienen el código fuente disponible. http://geotools.sourceforge.net/demos.php

En particular, podrías mirar el demo 'Village' (pueblo).

Además de esta guía de introducción, esperamos publicar guías adicionales sobre algunos temas específicos (como por ejemplo 'Highlight managers', 'Selection managers', 'Style objects' y 'Layer types' especiales).

Pero GeoTools no hace X

Si encuentras que, por mucho que estudias los ejemplos disponibles y la documentación API, no localizas una característica concreta que necesitas desesperadamente, tienes dos opciones posibles.

Primero, puedes enviarnos una solicitud de soporte. Siempre respondemos a las solicitudes de los usuarios si ésas son realistas, sobre todo si resultarían útiles para una amplia gama de usuarios. Desde luego, si puedes ofrecernos un incentivo financiero para desarrollar la característica requirida, podemos tratarlo como una prioridad urgente.

Segundo, puedes darte de alta en la lista de distribución geotools-discussion a ver si alguien ya esta desarrollando la característica que buscas o si están preparados a crearla (con o sin tu ayuda).

Por fin, desde luego, puedes arriesgarte tu mismo/a y desarrollar las características adicionales tu mismo/a. Si eres un programador competente, échale un vistazo a la guía de introducción para desarrolladores (disponible de momento sólo en inglés) y empieza.
